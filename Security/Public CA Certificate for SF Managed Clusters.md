# Managed Clusters using Public CA Certificates

## Solution Details

### Browser Error Issue

Service Fabric managed clusters (SFMC) use a cluster certificate to secure communication within the cluster. The cluster certificate is also used for TLS when serving the SFX application at the clusters management endpoint.

We are providing a new solution to address the issue of customers seeing untrusted browser messages when connecting to SFX.

Today the fqdn of the endpoint looks like this:

- `<clusterDnsName>.<region>.cloudapp.azure.com`
  - eg. `testcluster.southcentralus.cloudapp.azure.com`

As a result customers are faced with this error when connecting to Service Fabric Explorer:

![SFX Untrusted Error](..\..\images\sfx-untrusted.png)

With our proposed change it will look like:

- `<clusterResourceName>.<guid>.<region>.sfmc.io` 
  - Eg. `testcluster.adj35dskfope9rtk.southcentralus.sfmc.io`
  - To comply with Azure DNS requirements the new guid portion comes from the Azure DNS Deterministic Name Library and it is a unique identifier based off the cluster tenant, resource group, subscription and name

![SFX Trusted](..\..\images\trustedbrowser.jpg)

What this means:

- The cluster DNS will not be hosted by Public IP and will instead be hosted in a Service Fabric owned DNS Zone
  - This means that if any other service has a strong dependency on where the DNS entry is being hosted your availability will be affected while you switch traffic over
    - To address this issue we have added [functionality to break the migration into 2 steps](#what-to-do-if-you-depend-on-the-current-dns-name)

What does not change:

- Retrieving the Cluster FQDN
  - You can query the cluster resource through ARM and get the FQDN in the response

    ```powershell
    PS C:\Users\username> (Get-AzResource -ResourceId "/subscriptions/<subname>/resourcegroups/<rgname>/providers/Microsoft.ServiceFabric/managedClusters/<clustername>").Properties.fqdn
    
    clustername.h7bkh0bsaggpfdbn.centralus.sfmc.io
    ```
  
  - A second option would be to compute the FQDN using the same Azure DNS Deterministic Name Library (sample powershell tool can be found in official builds [pipeline artifacts](https://msazure.visualstudio.com/One/_build?definitionId=331649&_a=summary) (choose a recent build and look at the published artifacts) `drop_BuildAndTest_BuildAndTest\retail-amd64\CodeSign\Tools\DeterministicNameLibraryPowershell`) and the clusters ARM Resource ID
    - To use the powershell tool import the dll as follows: `ipmo .\DeterministicNameLibraryPowershell.dll`

    ```powershell
    cd C:\Users\username\Downloads\DeterministicNameLibraryPowershell
    ipmo .\DeterministicNameLibraryPowershell.dll
    $sub = New-Guid
    $tenant = New-Guid
    Get-SfmcDeterministicResourceName -Location southcentralus -AutoGeneratedDomainNameLabelScope SubscriptionReuse -TenantId $tenant -SubscriptionId $sub -ResourceGroupName "resourceGroupName" -ClusterName "clusterName" -Verbose
      VERBOSE: Begin!
      dhb0fbgwa3bbd7b4
    Get-SfmcDeterministicResourceName -Location southcentralus -AutoGeneratedDomainNameLabelScope SubscriptionReuse -TenantId $tenant -SubscriptionId $sub -ResourceGroupName "resourceGroupName" -ClusterName "clusterName" -GetFullSubjectName -Verbose
      VERBOSE: Begin!
      clusterName.dhb0fbgwa3bbd7b4.southcentralus.sfmc.io
    ```

### Cloud domain reuse issue

This is a compliance requirement for isolating OneCert registrations by cloud and environment. Currently, SFMC clusters use certificates issued under a shared domain: *.sfmc.azclient.ms. This domain is used across Azure Public, US and China clouds, which violates the isolation requirement. The Common Name (CN) in these certificates includes the domain (e.g., <guid>.sfmc.azclient.ms), making this a cross-cloud exposure. While this is not a direct Security risk - each certificate is cluster-specific and cannot be reused across clusters or clouds, SFMC already maintains independent Key and key vaults are isolated across regions - in order to be fully compliant. Customers can leverage our Public CA functionality which will allow customers to use a domain that is isolated to the environment they are creating in.

## How To Opt In

In order to create a cluster using the new format and certificate you must add [AutoGeneratedDomainNameLabelScope](https://learn.microsoft.com/en-us/azure/templates/microsoft.servicefabric/managedclusters?pivots=deployment-language-arm-template#managedclusterproperties-1) to your template.

This property specifies the reuse policy used in the Azure DNS Deterministic Name Library. For example, if you choose SubscriptionReuse, then using the same combination of ClusterName, Tenant and subscription id you will get the same guid hash regardless of the resource group used.

This property is introduced in the `2024-06-01-preview` api version.

```json
"type": "Microsoft.ServiceFabric/managedclusters",
"apiVersion": "2024-06-01-preview",
...
"properties": {
    ...
    "AutoGeneratedDomainNameLabelScope": "SubscriptionReuse"
},
```
### What to do if you depend on the current DNS name

For customers who have setups such as traffic managers or other DNS records pointing to the cluster DNS we are providing temporary functionality to help bridge the gap between these 2.

If you specify the tag **SFRP.PublicCertDefaultDnsOptIn** as follows
```json
"type": "Microsoft.ServiceFabric/managedclusters",
"apiVersion": "2024-06-01-preview",
"tags": {
  "SFRP.PublicCertDefaultDnsOptIn": "true"
}
...
"properties": {
    ...
    "AutoGeneratedDomainNameLabelScope": "SubscriptionReuse"
},
```

The Resource Provider(RP) will not remove the old dns entry from the public ip (`<clusterDnsName>.<region>.cloudapp.azure.com`). This will allow you to swap your CNAME record or equivalent from one DNS to the other, and you can swap back if things aren't working.

The expectation is that this is not a permanent state and that you should eventually migrate your workloads to use the new DNS entry that the RP is managing. Once you have been able to migrate away from using the `*.cloudapp.azure.com` DNS you can remove the tag from your resource and deploy your template to remove the old DNS entry.

## Migration

If you have an existing cluster that you want to use the new certificate configuration we have support to migrate the cluster in place. The migration is triggered through the same method as described in the [How To Opt In](#how-to-opt-in) section: add the property `AutoGeneratedDomainNameLabelScope` to your managedcluster definition.

### Important info

- This migration triggers 3 cluster upgrades and an extension upgrade on all the nodes to get the new certificate placed on all the nodes. A smaller cluster 6 vm instance non-zonal cluster (5 Upgrade Domains) takes just under 2 hours for the entire process - this is subject to other operations going on on the cluster. For cross AZ clusters (6-9 Upgrade domains) the time to upgrade will increase as upgrade domains increase.

- If the upgrade fails applying the new certificate on all the nodetypes please retry the deploying

- If you need to migrate back to an AME cert based cluster you can **keep the api version at "2024-06-01-preview"** and remove `AutoGeneratedDomainNameLabelScope` property and migration back will begin
  - This migration back will take just as long as it took to migrate forward

### Common Errors

#### System.TimeoutException: Operation 000000000-0000-0000-0000-000000000000 didn't complete within 0X:00:00

We have a timeout on waiting for the migration operation to finish. The longest successful migration we have seen thus far has been 7 hours though most migrations are much quicker. If you are hitting this timeout you might want to check if you are passing a customer UpgradeDescription to the managed cluster resource and if you are reevaluate the stable and wait durations in the monitoring policy as these extend how long the upgrades take.

Please retry the migration operation.

#### System.InvalidOperationException: Certificate was not applied on all nodes

This can mean that one of your nodetypes is in a bad state or there is another ongoing operation going on on this nodetype. If you see this error you can wait to retry the operation after a few hours. SF RP has a background worker that pulls new certs onto the machines and this can resolve the issue for you. Once the cert has be correctly pulled onto all nodes the migration will continue


## FAQ

1. Can my existing cluster migrate to this solution?

   To migrate an existing cluster follow the steps in the [How To Opt In](#how-to-opt-in) section.

2. Will the original cluster URL continue to work?

   Due to restrictions in naming that the original schema has we are replacing the old domain with this new format.

3. Does using this feature support Az Migration?

   Using the public ca cluster cert feature should not affect usability of az migration feature.

4. My cluster rapidly scales up and down. Will this affect this migration process?

   It is possible that constant scaling operations will pause the migration for too long and timeouts can occur. If you run into this it is recommended that you pause scaling on the cluster during the migration period. This is a one time operation and scaling can continue after the migration passes.

6. Will the migration flow affect my services? Will the upgrades respect my Maintenance Window?

   The 3 cluster upgrades operate as any other cluster upgrade – it will be UD by UD using health and safety checks – ie. the upgrade will ensure not to put services in situations where quorum cannot be made, if services go unhealthy this will block the upgrade. Workloads will get migrated off of nodes as needed, just as in other cluster upgrades like upgrading the SF Version or modifying the VMSS. As of now this migration is not affected by maintenance windows  as it is a customer initiated action and currently our maintenance window support is for background operations that the SF Control plane is initiating.

7. Will my App deployment get impacted by switching to use this feature?

   The only case where your Apps would get impacted is if they have a dependency on the dns name of the Cluster itself since that changes patterns from `<clusterDnsName>.<region>.cloudapp.azure.com` to `<clusterResourceName>.<guid>.<region>.sfmc.io` for Public regions.

8. What if I have a hard dependency on TrafficManager endpoints? How do things change?

   Traffic Manager allows to attach endpoints directly to a Public IP hosted DNS entry. However, Public IP does not support hosting custom DNS names. What they do support is a concept of [external endpoints](https://learn.microsoft.com/en-us/azure/traffic-manager/traffic-manager-endpoint-types#external-endpoints). Here you can add an endpoint that points to the Public IP Address of the cluster (this value doesnt change unless you do an ip hotswap) or point it to the new FQDN that you can precalculate using our [tool](#browser-error-issue). If you need both DNS entries around so that you can safely migrate from one entry to the other please look into our [provided solution](#what-to-do-if-you-depend-on-the-current-dns-name)
