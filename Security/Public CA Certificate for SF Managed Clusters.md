# Managed Clusters using Public CA Certificates

## Solution Details

### Browser Error Issue

Service Fabric managed clusters (SFMC) use a cluster certificate to secure communication within the cluster. The cluster certificate is also used for TLS when serving the SFX application at the clusters management endpoint.

We are providing a new solution to address the issue of customers seeing untrusted browser messages when connecting to SFX.

Today the fqdn of the endpoint looks like this:

- `<clusterDnsName>.<region>.cloudapp.azure.com`
  - eg. `testcluster.southcentralus.cloudapp.azure.com`

As a result customers are faced with this error when connecting to Service Fabric Explorer:

![SFX Untrusted Error](..\..\images\sfx-untrusted.png)

With our proposed change it will look like:

- `<clusterResourceName>.<guid>.<region>.sfmc.io` 
  - Eg. `testcluster.adj35dskfope9rtk.southcentralus.sfmc.io`
  - To comply with Azure DNS requirements the new guid portion comes from the Azure DNS Deterministic Name Library and it is a unique identifier based off the cluster tenant, resource group, subscription and name

![SFX Trusted](..\..\images\trustedbrowser.jpg)

What this means:

- The cluster DNS will not be hosted by Public IP and will instead be hosted in a Service Fabric owned DNS Zone
  - This means that if any other service has a strong dependency on where the DNS entry is being hosted your availability will be affected while you switch traffic over
    - To address this issue we have added [functionality to break the migration into 2 steps](#what-to-do-if-you-depend-on-the-current-dns-name)

What does not change:

- Retrieving the Cluster FQDN
  - You can query the cluster resource through ARM and get the FQDN in the response

    ```powershell
    PS C:\Users\username> (Get-AzResource -ResourceId "/subscriptions/<subname>/resourcegroups/<rgname>/providers/Microsoft.ServiceFabric/managedClusters/<clustername>").Properties.fqdn
    
    clustername.h7bkh0bsaggpfdbn.centralus.sfmc.io
    ```
  
  - A second option would be to compute the FQDN using the same Azure DNS Deterministic Name Library (sample powershell tool can be found in official builds [pipeline artifacts](https://msazure.visualstudio.com/One/_build?definitionId=331649&_a=summary) (choose a recent build and look at the published artifacts) `drop_BuildAndTest_BuildAndTest\retail-amd64\CodeSign\Tools\DeterministicNameLibraryPowershell`) and the clusters ARM Resource ID
    - To use the powershell tool import the dll as follows: `ipmo .\DeterministicNameLibraryPowershell.dll`

    ```powershell
    cd C:\Users\username\Downloads\DeterministicNameLibraryPowershell
    ipmo .\DeterministicNameLibraryPowershell.dll
    $sub = New-Guid
    $tenant = New-Guid
    Get-SfmcDeterministicResourceName -Location southcentralus -AutoGeneratedDomainNameLabelScope SubscriptionReuse -TenantId $tenant -SubscriptionId $sub -ResourceGroupName "resourceGroupName" -ClusterName "clusterName" -Verbose
      VERBOSE: Begin!
      dhb0fbgwa3bbd7b4
    Get-SfmcDeterministicResourceName -Location southcentralus -AutoGeneratedDomainNameLabelScope SubscriptionReuse -TenantId $tenant -SubscriptionId $sub -ResourceGroupName "resourceGroupName" -ClusterName "clusterName" -GetFullSubjectName -Verbose
      VERBOSE: Begin!
      clusterName.dhb0fbgwa3bbd7b4.southcentralus.sfmc.io
    ```
